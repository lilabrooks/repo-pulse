- Proposed scope change + README provenance (docs only, no code): recorded proposed ADR 0003 (homepage-to-repo resolution) for review — mapping non-GitHub URLs like `lenshq.io/products/lens-k8s-ide` to a repo, recommending the GitHub Search API behind user confirmation, and flagging that it needs a `docs/GOAL.md` constraint change (owner-reserved); not implemented. - Frontend input parsing (post-milestone usability fix): the form was rejecting anything but bare `owner/repo` — the HTML `pattern` attribute (one slash) plus a strict two-segment `split("/")` blocked pasted GitHub URLs before any fetch. Reported case: pasting `lenshq.io/products/lens-k8s-ide`. Added a `parseRepo()` helper (`static/app.js`) that accepts `owner/repo`, `github.com/owner/repo`, and full `https://github.com/owner/repo[/...]` URLs (stripping scheme, `www.`, host, `.git`, and any trailing path/query/fragment), removed the stale `pattern` attribute, and updated the placeholder/aria-label. Non-GitHub hosts are rejected with a clear message rather than silently mis-parsed, since the goal fixes external data to the GitHub REST API only — so `lenshq.io/...` (a product homepage, not a GitHub URL, no REST-derivable owner/repo) gets a helpful error, not a confusing 404. `docs/specs/frontend.md` updated to describe the accepted forms. No backend/API change. Verified with a Node DOM-stub harness driving the real submit handler (7 wiring cases: URL/host/.git/bare/space-encoding resolve to the right encoded fetch; non-GitHub and single-segment inputs fetch nothing and show an error; 404 detail shown verbatim) plus a 18-case pure-logic table, and live against the running server (served `app.js` carries `parseRepo`, `/` has no `pattern`, `github.com`-style query resolves through to the GitHub API). JS still has no automated harness in `make test` per the no-Node constraint; these checks were throwaway scripts.
